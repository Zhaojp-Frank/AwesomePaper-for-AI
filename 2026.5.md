## SparseMoE
Uncovering Intra-expert Activation Sparsity for Efficient Mixture-of-Expert Model Execution

https://arxiv.org/pdf/2605.08575 2026.5.9 伯克利
paper里有源码 sparse MoE execution kernel

1.  探讨了MoE模型中未被充分利用的**intra-expert activation sparsity**，发现现有预训练模型在不牺牲显著准确性的情况下，**专家内激活可达高达90%的稀疏度**。
2.  扩展vLLM的MoE执行管线，实现了对**非活跃神经元计算的跳过**，从而在**MoE层获得了2.5x的速度提升**，并实现了端到端**1.2x的推理加速**。
3.  揭示了**共享专家神经元对模型准确性的影响远大于路由专家神经元**，为高效MoE模型执行提供了新的实用优化维度。

关键观测：Qwen3.5/DeepSeekv2/GPT-OSS等模型 token选中某专家，只需要激活该专家的一小部分神经元，通过对激活得分较低的神经元进行**零化处理**，在**不显著损失精度的情况下，每个专家内部可实现90%的稀疏度**。大尺寸下，85%～90%的稀疏度；模型小 稀疏%比较低（50%）
batch中大（>16）加速更大（MoE layer +1.5x ~2x,H200）;4090加速更大（～2.5x）

**场景与具体问题：**
MoE 架构因其通过稀疏激活专家实现的计算效率，已成为当前最先进 LLMs 的标准。然而，通过更细粒度的专家（即增加专家数量并减少每个令牌激活的专家比例）来进一步提高稀疏性面临着根本性的训练挑战，例如专家崩溃（expert collapse）、负载不平衡（load imbalance）、表示崩溃（representation collapse）以及每个专家梯度更新减少等问题。这些挑战随着专家数量的增加而加剧，使得进一步提升专家间稀疏性变得困难。本文提出在推理时利用每个专家内部激活的稀疏性来解决这一问题。

**业界存在不足：**
1.  现有关于激活稀疏性的研究通常针对较旧的密集型架构或需要重新训练才能实现稀疏性，这极大地限制了其对广泛使用的预训练 MoE 模型的适用性和实用性。
2.  缺乏将专家内部激活稀疏性端到端集成到高度优化的生产级执行系统（如 vLLM）中的实践。现有研究多停留在朴素的实验层面，未能评估该技术在高性能系统中的兼容性和实际效益。

**关键观察与假设：**
*   **假设：** 当一个令牌被路由到一个专家时，它可能只需要激活该专家的一小部分神经元，而不是整个专家。
*   **关键发现（令人惊讶）：** 在现有的预训练 MoE 模型中，无需修改激活函数或模型参数，就已存在大量的专家内部激活稀疏性。通过对激活得分较低的神经元进行零化处理，在不显著损失精度的情况下，每个专家内部可实现高达 90% 的稀疏度。
*   研究还发现，共享专家（shared experts）和路由专家（routed experts）中的神经元对模型精度的影响显著不同：共享专家神经元的影响更大。对于大型模型，在所有共享专家神经元都活跃的情况下，即使路由专家仅激活 2% 的参数也足以维持模型容量。

**方法核心思路和主要步骤：**
论文的核心方法是对 vLLM 的 MoE 执行管道进行重新设计，以高效利用专家内部激活稀疏性。

1.  **稀疏性量化与阈值确定：**
    *   通过对预训练 MoE LLMs 进行基准测试，衡量不同专家内部激活稀疏度水平下的模型得分。稀疏性通过对每个专家的激活输出进行排序并零化最低得分的神经元来实现。
    *   将用户指定的稀疏度转换为模型特定的神经元去激活阈值。如果神经元的绝对激活输出值低于此阈值，则该神经元被屏蔽（masked out）。这实现了 $O(n)$ 的稀疏掩码生成，而非传统的 $O(n \log n)$ 顶部-k 方法，且精度损失极小。
2.  **vLLM 管道修改（路由专家）：**
    *   仅修改路由专家的执行代码路径，保持共享专家执行、路由器（router）、分派（dispatch）和组合（combine）功能不变。共享专家路径不变是因为它们与路由专家路径并行执行且神经元对精度影响大。
    *   **独立 Gate 投影：** Gate 投影必须在所有稀疏操作之前执行，因为其输出是计算神经元激活和稀疏掩码所必需的。为此，复用用于密集路径下投影的批处理矩阵乘法（BMM）核函数。这意味着稀疏计算仅限于约三分之二的执行（Up 和 Down 投影）。
    *   **自定义稀疏激活核函数：** 使用 Triton 实现一个自定义的稀疏激活核函数，该函数将激活计算和失活神经元的掩码操作融合在一个核函数中。该核函数根据 Gate 投影输出计算神经元激活，并应用基于阈值的单次掩码。
    *   **融合稀疏 Up-Down 投影核函数：** 基于 Triton BMM 核函数，创建一个融合的稀疏 Up-Down 投影核函数。它首先使用批处理向量加载（batched vector load）高效收集给定输入令牌的每个激活神经元的 Up 权重向量，形成一个紧凑的、密集的 Up 权重矩阵。该矩阵与输入令牌进行密集矩阵向量乘法得到 Up 投影输出。输出再乘以激活值，作为与 Down 投影权重进行矩阵向量乘法的输入，Down 权重也通过批处理向量加载进行收集和打包。
    *   **兼容性与优化：** 为保持与 HIP Graph 和 CUDA Graph 的兼容性，确保 GPU 内存缓冲区和核函数启动参数在图调用中复用。激活神经元索引缓冲区被填充到完整的神经元大小，Triton 核函数以固定的 64 神经元平铺粒度启动。
    *   **混合执行策略：** 在引擎启动时分析稀疏执行路径和密集执行路径之间的“临界批处理大小”（tipping batch size），在运行时根据当前批处理大小动态切换使用稀疏或密集执行路径，确保在不同负载下都能获得性能提升。

**实验设置：**
*   **实现：** 基于 vLLM 的开发版本（version 0.1.dev15965+g6506aec547）。
*   **硬件：** AMD MI355X、NVIDIA H200 和 NVIDIA RTX4090 GPU。
*   **模型：** Qwen3.5-35B-A3B 和 Qwen3.5-122B-A10B。
*   **数据：** WikiText-2 数据集样本。
*   **负载：** 实验涵盖了 60%、70%、80%、85% 和 87% 的总激活稀疏度水平，以及不同的批处理大小和请求数量。
*   **对比基线：** 原始的密集 vLLM 基线。

**关键对比结果：**
*   **MoE 层性能：** 在 MoE 层执行中实现了高达 1.5–2.5 倍的加速，其中 Qwen3.5-35B-A3B 在 85% 和 87% 稀疏度下实现了 1.55 倍的加速。加速效果强烈依赖于稀疏度和批处理大小，在批处理大小为 16 到 128 时达到峰值。硬件资源受限越严重，加速效果越明显：RTX4090 实现的加速最高（2.5 倍），MI355X 较低（1.8 倍）。更大的模型（Qwen3.5-122B-A10B）也显示出更高的加速。
*   **稀疏度与精度：** 阈值化的激活稀疏 MoE 执行在目标稀疏度下实现了与基线平均基准分数 95% 以上的准确率保持，这与早期分析中的模型稀疏度截止值吻合。
*   **执行时间分解：** 密集 Gate 投影和激活核函数贡献了恒定的执行开销，只有基于 Gather 的融合 Up-Down 投影核函数受益于稀疏度增加。
*   **端到端性能：** 考虑到整个 vLLM 系统的开销，端到端推理速度提升高达 1.2 倍（在 H200 上）和 1.1 倍（在 MI355X 上）。加速主要体现在延迟敏感型工作负载（如大量输出令牌或少量请求的解码密集型任务）。由于阿姆达尔定律，尽管 MoE 层加速显著，但由于其他组件（注意力、嵌入层、vLLM 调度开销）的存在，总体的端到端加速被限制。

**潜在局限或不足：**
1.  **Gate 投影的稀疏性限制：** 当前设计仅将激活稀疏性应用于 Up 和 Down 投影，这仅占专家 FFN 计算的大约三分之二。这导致端到端加速的理论上限为 3 倍。如果能实现轻量级的 Gate 激活预测机制，或者重用基于输入令牌的稀疏模式，以避免完整的密集 Gate 计算，将能进一步提升性能。
2.  **组内专家激活稀疏性粒度优化：** 稀疏计算在 GPU 硬件上的高效映射引入了开销（不规则内存访问、索引、硬件利用率受限）。随着批处理大小的增加，理论稀疏性和实际加速之间的差距增大。未来工作可以探索增加稀疏性映射的粒度，将令牌和神经元分组以更好地匹配硬件加速器宽度，在块级别执行稀疏性映射，从而提高内存合并和硬件利用率。
   
## FCP
Unleashing Scalable Context Parallelism for Foundation Models Pre-Training via FCP
https://arxiv.org/pdf/2605.08575 2026.5.9 

https://arxiv.org/pdf/2605.08524 2026.5.8 字节

1. 针对基础模型**预训练中多样化的上下文长度序列**，现有Context Parallelism (CP) 方法因过度分片短序列导致计算效率低下，或因长短序列处理隔离导致负载不平衡。
2. FCP提出一种**灵活的上下文并行**范式，通过将**序列分片为固定大小的块**，并采用工作**负载感知的块分配器和无拥塞的通信规划器**，实现了块级别的调度和任意对等通信。
3. Llama-3-70B 256卡（A/H?）对比RingAttn，MegiAttn，ByteScale等，并将attention MFU提升了1.13倍至2.21倍，显著优于现有CP框架。

FCP论文旨在解决基础模型预训练中长上下文序列带来的可伸缩性挑战，特别是在现实世界数据集中上下文长度高度多样化的背景下。

**场景与具体问题**
基础模型训练中，长上下文序列（如文档、多模态输入）导致自注意力模块的内存和计算需求激增。注意力计算复杂度为$O(HDL^2)$，内存复杂度为$O(HLD)$，其中$L$是序列长度。上下文并行（Context Parallelism, CP）被提出，通过分片序列到多个GPU上以降低单GPU内存使用并提升计算能力。然而，现实世界语料库的上下文长度呈现出高度多样性（长尾分布，如图2所示，最长达512K），对高效CP算法设计构成显著挑战。主要问题体现在两方面：
1.  **计算效率低下 (Compute Inefficiency)**：现代GPU（如TensorCores）需要大输入才能饱和。现有CP方法常将短序列过度分片成过小的块（例如小于2K tokens），导致硬件利用率低下（如图3所示，MFU在len(B) < 2K时显著降低）。更糟的是，这些微小分片仍需跨GPU传输，造成过量通信，进一步降低计算效率。例如，约50%的通信量在朴素CP中来自长度小于4K的序列，而这些序列本可不分片。
2.  **工作负载不平衡 (Workload Imbalance)**：注意力计算量随上下文长度$L$呈二次方增长。即使每个GPU分配的总tokens数量相同，上下文长度分布不均也会导致显著不平衡，部分GPU过载而其他空闲，降低集群整体效率。内存使用和通信量也需平衡。

**业界存在哪些不足**
现有CP设计通过简化调度问题来降低搜索空间，导致次优性能：
*   **平衡优化 (Balance-optimized)**：如Ring Attention。它统一将所有序列分片为固定数量的块（例如$2N$个），并使用Zig-Zag顺序分配以实现完美的工作负载平衡。然而，它忽略了上下文长度和计算效率$f(\cdot)$，导致短序列过度分片成小块，严重降低MFU（如图1左，图5a）。
*   **效率优化 (Efficiency-optimized)**：如ByteScale。它根据上下文长度将序列分组，并按比例分配不同数量的worker。这提高了计算效率，但由于在不同GPU上隔离了子组，导致工作负载严重不平衡，特别是对于极端长度的异常值序列（如图1中，图5b）。
*   **混合方法 (Adaptive switching)**：如WLB-LLM。它根据性能评估器在上述两种方法之间切换，但并未扩展有限的搜索空间，本质上仍是计算效率和负载平衡之间的权衡。
*   **刚性拓扑 (Rigid Ring Topology)**：所有现有设计都依赖环形拓扑（无论是单一环还是异构子环），这简化了通信模式，但也极大地限制了分片放置的灵活性，使得CP调度从根本上是次优的。

**关键观察与假设**
*   通信开销可以通过计算/通信重叠来有效缓解。在现代GPU和网络条件下，例如Hopper GPU与50GB/s InfiniBand网络，仅需22GB/s（44%带宽）即可实现完全重叠（$\eta=1$）。
*   增加块大小len(B)可以降低带宽要求，因为块内计算量随块大小二次方增长，而通信量线性增长，这提供了调度粒度与通信效率之间的灵活权衡。
*   通过允许任意点对点通信，消除环形拓扑的约束，可以实现理想的工作负载平衡。

**方法核心思路和主要步骤 (FCP设计)**
FCP（Flexible Context Parallelism）是一个新的CP范式，通过**块级别粒度的分片和灵活的GPU分配**，实现注意力计算的线性可伸缩性。
1.  **块分发器 (Block Distributor)**：
    *   **分片策略 (Sharding Policy)**：将每个序列分解成固定大小的块（例如1K tokens）。这大大减小了最优调度的搜索空间，同时保持表达能力（长序列块多，短序列块少）。块大小len(B)的选择考虑硬件饱和度（如图3），以及实现计算与通信重叠所需的网络配置（§3.5）。对于小于块大小的序列，FCP会将其打包到最少数量的块中，并使用变长注意力API进行计算。
    *   **分配策略 (Assignment Policy)**：采用LPT（最长处理时间优先）调度算法的变体，将分片后的块分配给GPU。它贪婪地将每个块分配给当前负载最低的worker。负载定义为内存和计算使用量的加权和。目标是最小化所有worker中的最大计算负载，同时满足每个worker的内存限制（算法1）。时间复杂度为$O(K \log N)$，可忽略不计。

2.  **通信规划器 (Communication Planner)**：
    *   **块级流水线 (Block-level Pipelining)**：将跨worker的计算和通信分解为块级子阶段，并交错执行（如图7）。CP通常包括拉取远端块、计算注意力块、推送本地块三个阶段。FCP将每个阶段分解为细粒度的块级子阶段，并逐块交错，实现通信与计算的有效重叠。
    *   **无拥塞求解器 (Congestion-free Solver)**：为避免通信拥塞（例如多个worker同时从同一worker拉取块），FCP将worker间的数据流建模为二分图。图中有$N$个发送节点和$N$个接收节点，边$S(i) \to R(j)$表示块从GPU $i$传输到GPU $j$。一个单阶段无拥塞通信定义为匹配（每个worker最多发送/接收一个块）。FCP利用霍普克罗夫特-卡普算法（Hopcroft-Karp algorithm），以$O(\hat{N}^{2.5})$的复杂度迭代计算最大匹配作为块级通信计划，确保了块传输的最优顺序和无拥塞（如图8）。最小无拥塞子阶段数量等于最大度$\Delta$，FCP能将所有边划分为$\Delta$个不相交的匹配。
    *   **自底向上合并器 (Bottom-up Coalescer)**：将细粒度的块级子阶段合并成粗粒度阶段（例如，将连续4个子阶段合并），而不影响无拥塞通信特性。这通过增加执行块的大小来提高内核效率（§3.1），本质上解耦了调度粒度（块大小）和执行粒度（合并后的块大小），提供了更大的灵活性。

3.  **透明重排器 (Transparent Reshuffler)**：
    *   为确保FCP能透明地集成到现有框架中，FCP在进入注意力模块时，将用户提供的序列布局重排为CP所需的负载感知布局，并在退出后恢复。这种重排操作被集成到块级流水线中，并与本地注意力计算重叠，以最小化开销。本地计算（不依赖远端块）被优先执行。

**实现细节**
*   FCP核心代码约4K行Python，基于FlashAttention3内核进行少量修改。
*   利用CUDA Green Contexts将GPU SMs空间分区为通信和计算专用，避免干扰。
*   通信使用NCCL的组点对点原语，并感知网络拓扑（例如，为rail-optimized网络启用Ulysses避免两跳通信）。
*   块级流水线采用多缓冲区（默认3个缓冲区）以并发启动通信操作，减少流水线气泡。

**实验设置**
*   **硬件**：在工业界和学术界的两个计算集群上进行评估，使用GPU-X和GPU-Y两种GPU类型。GPU-X的计算/通信比为5920，GPU-Y为2500。集群规模高达256块NVIDIA GPU。
*   **模型与工作负载**：采用Llama-3-70B模型配置（8个KV头，64个QO头，头维度128）。序列长度从图2所示的真实训练轨迹中随机采样，最大序列长度512K。每GPU token数量固定为32K（在§6.6中测试不同数量）。应用因果注意力掩码。
*   **对比基线**：
    1.  **Ring Attention**：平衡优化设计。
    2.  **ByteScale**：计算效率优化设计，动态将短序列和长序列分配到不同GPU。
    3.  **WLB-LLM**：基于在线性能估算器在平衡优化和效率优化基线之间自适应切换。

## PPD turn2

https://arxiv.org/pdf/2603.13358v1 2026.3.9 TogetherAI

**场景与具体问题**
LLM推理中，PD分离已成为标准架构，它将计算密集型预填充（Prefill）和内存密集型解码（Decode）阶段部署在独立的GPU池上，以缓解两种不同工作负载的干扰。然而，随着聊天机器人和代理系统对多轮交互需求的增长，传统的PD架构在这种场景下暴露出两个根本性低效：
1.  **重复预填充与重复KV计算：** 每次新一轮交互，即使只有少量新输入，也需要重新预填充上回合的响应和新提示，导致KV缓存的重复计算，尤其是在长上下文场景下，高达99%的预填充成本来源于历史Token的KV重计算。
2.  **KV传输饱和：** 预填充节点计算出的KV缓存需要通过网络传输到解码节点，多轮交互中重复的KV传输会饱和网络带宽，导致高延迟甚至服务降级，特别是在高负载下。这使得第二轮及后续轮次（Turn 2+）的首次Token时间（TTFT）居高不下，即使输入较短。

**业界存在哪些不足**
虽然PD分离有效隔离了预填充和解码工作负载，解决了单轮推理中的干扰问题，但其核心痛点是每次请求（包括后续轮次）都需要通过网络传输完整的KV缓存。现有方法如分布式KV缓存（CachedAttention, Mooncake, MemServe, LMCache）或前缀缓存（vLLM, SGLang）主要集中于外部KV缓存存储和复用，或要求请求返回到同一实例，但并未从根本上解决“暖”请求（即已缓存KV状态的请求）在解码节点本地处理的路由效率问题。此外，尽管有并发工作（如AMPD）也提出将增量预填充路由到解码节点，但它们更多依赖于在线队列状态估计，缺乏对微架构干扰的严谨分析和基于离线优化的路由决策。

**关键观察与假设**
本研究的核心洞察在于：并非所有预填充操作都具有相同的干扰性。通过微架构干扰分析，作者发现**追加预填充（Append-Prefill，AP）**——仅处理新的输入Token同时复用已缓存的KV状态——比**完全预填充（Full Prefill）**——处理没有缓存上下文的新提示——对解码速度的影响要小得多（一个数量级）。具体而言，完全预填充会导致显著的解码减速（单并发预填充时减速约48%，四并发时约57%），而追加预填充的干扰则非常小（单并发时仅减速约2%，四并发时约21%），并且其干扰在面对更高并发和更长上下文时（即使在64K Token下也低于25%）仍然保持在可控范围内。这一观察推翻了“所有预填充操作都严重干扰解码”的假设，并引出核心假设：将追加预填充操作本地路由到解码节点是可行的且高效的。

**方法核心思路和主要步骤**
PPD（Prefill Prefill-capable Decode）是一种动态路由系统，旨在解决“追加预填充操作应在预填充节点还是解码节点上计算”这一核心问题。PPD通过一个优化问题来形式化多轮推理服务，其目标是优化用户定义的服务水平目标（SLO），如TTFT或每输出Token时间（TPOT），并动态决定何时将Turn 2+请求本地路由到解码节点。PPD认为，传统的PD模式（即所有AP操作都发送到预填充节点）是该优化问题中 $\text{x}=0$ 的一个特例。

PPD的核心思想是根据当前工作负载估计、用户指定的SLO和初始节点分配来动态调整将追加预填充操作路由到解码节点的百分比 $\text{x} \in [0, 1]$。它分两个阶段运行：

1.  **阶段1：离线表格构建（Phase 1: Offline Table Construction）**
    *   在离线阶段，PPD会对预定义的工作负载网格（包括不同的上下文长度、输入/输出长度、QPS等）进行基准测试。
    *   对于每个离散化的工作负载 $\hat{\psi}_i$，分别测量在两种极端路由策略下的Turn 2+ TTFT和TPOT：
        *   $\text{x}=0$：所有AP操作路由到预填充节点（传统PD）。
        *   $\text{x}=1$：所有AP操作路由到解码节点（Full AP-to-D，即本地处理）。
    *   计算两种策略下TTFT的相对改进率 $\Delta_{ttft}$ 和TPOT的相对恶化率 $\Delta_{tpot}$。
    *   根据用户定义的权重 $w_{ttft}$ 和 $w_{tpot}$ 计算一个加权分数：$\text{score} = w_{ttft} \cdot \Delta_{ttft} - w_{tpot} \cdot \Delta_{tpot}$。
    *   如果 $\text{score} > 0$，则将该工作负载的最佳路由策略标记为 $\text{x}^*(\hat{\psi}_i) = 1$（路由到解码节点）；否则标记为 $\text{x}^*(\hat{\psi}_i) = 0$（路由到预填充节点）。
    *   这个过程生成一个预计算的决策表，存储了在不同工作负载条件下，将AP操作路由到解码节点是否更优的决策。

2.  **阶段2：在线每请求决策（Phase 2: Online Per-Request Decision）**
    *   当一个Turn 2+请求到来时，PPD路由器会提取其工作负载特征（如当前Turn数、输入Token数、输出Token数、上下文长度和QPS）。
    *   通过离散化，将当前请求的工作负载映射到离线阶段构建的决策表中最近似的工作负载条目 $\hat{\psi}_i$。
    *   根据决策表中预计算的 $\text{x}^*(\hat{\psi}_i)$ 值，决定该请求的追加预填充操作是路由到预填充节点还是解码节点。
    *   对于Turn 1请求，由于没有缓存上下文，始终执行 $\text{x}=0$ 策略（即所有KV传输都通过P节点进行）。
    *   这个决策查找过程开销极小（每次请求小于1毫秒）。

**实验设置**
*   **实现基础：** PPD原型基于vLLM的分布式服务基础设施构建，复用了其KV传输协议和前缀缓存机制。路由代理使用Quart（异步Flask）实现。
*   **硬件条件：** 实验在配备4块NVIDIA H100 80GB HBM3 GPU的内部集群节点上进行，GPU之间通过NVLink连接。
*   **软件版本/模型：** 主要模型为Llama-3.1-8B，并通过Qwen2.5-14B和Qwen3-30B-A3B进行了验证性实验。
*   **负载情况：**
    *   **配置空间：** 评估了17种GPU配置，重点关注10种核心配置，包括Replica（4R）、传统PD（x=0，如1P 3D, 2P 2D, 3P 1D）、Full AP-to-D（x=1，同上）和部分AP路由（0<x<1，如x=1/3, 1/2, 2/3）。
    *   **工作负载：**
        *   **合成工作负载：** 设计了18种合成工作负载，结合2种Turn-1配置文件和9种Turn-2配置文件（涵盖解码密集型、平衡型和预填充密集型），以覆盖多轮对话模式的完整范围。
        *   **真实世界工作负载：** 使用两个公开的多轮对话数据集：ShareGPT（用户共享的ChatGPT对话）和WildChat（野外用户对话）。
    *   **QPS水平：** 每个配置在10个不同的QPS水平（0.5, 1, 2, 4, 6, 8, 10, 12, 16, 20）下进行评估。
    *   **对话长度：** 验证了2-16轮对话的场景。
*   **对比基线：**
    *   **传统PD（x=0）：** 所有后续轮次请求都发送到预填充节点并进行KV传输。
    *   **Replica：** 所有操作都在单个GPU上本地执行。
    *   **Full AP-to-D（x=1）：** 所有后续轮次请求都在解码节点本地处理。
    *   **固定部分AP路由（0<x<1）：** 特定比例的AP请求路由到D节点。
    *   **PPD：** 动态路由系统，根据SLO和工作负载决定x值。

**关键对比结果**
*   **TTFT显著降低：** PPD将Turn 2+的TTFT降低了48%至73%（相对于传统PD），在ShareGPT数据集上，将平均查询延迟降低了15%至25%。这种改进主要通过消除后续轮次的KV传输实现。
*   **TPOT保持竞争力：** PPD在实现TTFT显著降低的同时，对TPOT的影响极小，TPOT恶化范围控制在7%至12%以内。通过调节权重参数 $w_{ttft}$ 和 $w_{tpot}$，操作员可以在TTFT优化和TPOT保护之间进行平滑插值。
*   **吞吐量提升：** PPD通过KV缓存复用和减少网络负载，提高了系统效率，在吞吐量方面表现优异，最高可达到38.3%的获胜率。
*   **服务稳定性提升：** 传统PD在高QPS下由于KV传输饱和会导致服务降级（成功率<95%），而PPD在所有QPS水平和数据集上均实现了100%的成功率，将先前不可用的配置转变为稳定的部署。PPD消除了网络瓶颈，将KV传输量减少了约75%。
*   **动态路由的必要性：** 实验结果显示，没有任何单一的固定 $\text{x}$ 值在所有SLO（TTFT、TPOT、吞吐量）上都占据主导地位。92.2%的工作负载-QPS组合在T2 TTFT和平均TPOT方面有不同的最优配置，证实了动态路由的必要性。PPD在所有目标上都达到了最佳的帕累托前沿。
*   **可扩展性：** PPD的优势在2-16轮对话和8B-30B模型尺寸上都保持稳定，TTFT改进约70%，表明其益处源于架构特性而非模型特异性。

**潜在局限或不足**
*   **与分布式KV缓存的关系：** 尽管PPD与分布式KV缓存是互补的，但PPD本身并不直接提供集群范围内的KV缓存存储和共享机制。它侧重于调度和路由，而非存储层。
*   **实时队列状态：** PPD采用离线预计算的决策表，而非像AMPD那样基于实时队列延迟估计进行决策。这虽然带来了可预测性和低运行时开销，但在面对快速变化且预测性差的瞬时负载尖峰时，可能不如基于实时感知的调度灵活。
*   **实验规模：** 尽管在H100 GPU上进行了广泛测试，但主要模型（Llama-3.1-8B）的选择部分是出于实用性考虑，因为测试更大模型的17种配置组合会带来巨大的初始化开销。这可能限制了结果向超大型模型（如数百亿或万亿参数模型）的直接泛化。
*   **非纯P/D配置：** 论文排除了混合配置（如R与P/D组合）的深入分析，认为它们通常性能不佳且与AP路由优化正交。然而，在某些边缘场景下，混合配置可能仍有其考量价值。

**场景与具体问题**
Mixture-of-Experts (MoE) 架构通过稀疏激活专家（FFN层），实现了万亿参数级别LLM的训练，降低了每个token的计算量。然而，EP作为MoE训练的常用策略，需要将专家分片到不同的GPU设备上，而密集组件如注意力层则通过数据并行（DP）复制。这导致了两次关键的all-to-all通信：token从注意力层的微批次排序“dispatch”到专家层的专家排序，然后“combine”回来。这些通信位于注意力层和MoE层之间的关键路径上，并传输几乎整个微批次的数据。随着模型规模的增长，FFN层无法再适应单个多GPU节点，EP域需要跨越多个节点，此时较低的节点间带宽（如InfiniBand相比NVLink）进一步放大了通信瓶颈。论文剖析发现，在64张H800 GPU上，这些通信可能占据训练步骤高达50%的时间（图1），并且随着EP跨越更多节点，all-to-all通信占比从单节点的22%上升到8节点的78%（图3a）。

**业界存在哪些不足**
1.  **操作级重叠的局限性：** 现有工作（如Tutel、Comet）尝试通过操作级切分来重叠all-to-all通信与FFN计算（图4b）。然而，这种重叠窗口受限于FFN计算量，通常不足以完全隐藏通信，导致残余的网络I/O停滞（图5）。FlashAttention等技术使得自注意力计算难以完全切分，进一步限制了重叠效果。
2.  **微批次级流水线的不足：** Lancet和DualPipe通过流水线化多个微批次来扩大重叠窗口（图4c）。但它们面临根本性的计算-通信失衡：注意力计算时间随序列长度二次方增长，而FFN计算和all-to-all通信大致呈线性增长。这种不平衡使得在不同工作负载下难以实现完美重叠（图6a），导致即使在稳态下也存在未重叠的尾部。
3.  **现有注意力-FFN分离（AFD）方法的挑战：** 尽管有工作（如MegaScale-Infer、StepFun、HeterMoE）尝试将注意力和FFN层分离到不同的设备组，但在训练场景下存在问题。每个组必须持有其组件在所有层中的全部参数以及训练状态（如优化器），导致在扩展到千亿参数模型时易发生内存溢出（HeterMoE仅在约4.3B参数规模下验证，图14），使得如何为大规模MoE训练实现通信重叠成为开放问题。此外，现有方法保持固定的计算-通信比，限制了平衡通信和计算的设计空间。

**关键观察与假设**
1.  **计算-通信特性差异：** 论文观察到注意力组件（A）和FFN组件（F）的计算-通信特性存在显著差异。注意力FLOPs随序列长度二次方增长，导致其计算-通信比（算术强度）增长更快，更易达到计算瓶颈。而FFN计算和通信量均呈线性增长，其计算-通信比相对稳定，更容易受通信带宽限制。在EP组内，注意力高效利用计算资源，而FFN受通信限制（图6b）。
2.  ** disaggregated 架构的潜力：** 这种差异表明，将注意力层和FFN层分离到独立资源组（即 disaggregated attention-FFN, AFD）可以允许每个组件在其最优计算-通信平衡点附近运行。通过自适应地分配计算和网络资源，可以有效提升整体系统性能（图9）。
3.  **训练负载的稳定性：** 预训练工作负载通常具有固定的序列长度和批次大小，这意味着可以进行一次性的静态资源分配来平衡各阶段的吞吐量和利用率。

**方法核心思路和主要步骤**
DisagMoE 提出了一种名为 AF-Pipe 的多阶段流水线并行策略，并结合了自适应工作节点分配。

1.  **解耦组件放置（Disaggregated Component Placement）：**
    *   模型按组件类型划分：注意力层分配给A-Workers组，FFN层分配给F-Workers组。
    *   A-Workers内部通过数据并行（DP）复制密集组件和分发微批次训练数据。
    *   F-Workers内部通过专家并行（EP）分片专家并处理token。
    *   每个组件类型可以有 $p$ 个组，每个组负责该类型组件的 $1/p$ 子集，例如，层 $L_g^c = \{g + kp \mid k = 0, 1, \dots, v-1 \}$。这种设计确保了组的同质性，并最大化了设备利用率，同时在固定设备内存预算下支持更大的MoE配置。

2.  **AF-Pipe 流水线与重叠：**
    *   **时间复用调度：** AF-Pipe 是一种为 disaggregated 架构定制的流水线并行（PP）策略。它将批次划分为微批次，并在微批次级别进行流水线执行。前向传播时，中间结果（隐状态）从A-Workers传输到F-Workers；反向传播时，梯度从F-Workers传回A-Workers。这确保了两个组件组的持续活跃，提高效率。
    *   **M2N 阶段边界：** 传统的all-to-all通信被替换为 A-Workers 和 F-Workers 组之间的 many-to-many (M2N/N2M) 通信原语。M2N通信是全双工的，可以充分利用带宽。通过将常规的点对点（P2P）和合并操作融合到一个统一的通信阶段中，AF-Pipe 将通信成本降低了约 $1/k$ (k为top-k专家数)，并实现了计算与通信的系统性重叠。
    *   **异步执行流：** 每个A-组和F-组维护专用的发送和接收 ProcessGroups，并在各自工作节点内的三个协调流（前向、后向和通信）中异步启动，以保证非阻塞执行。例如，在稳态下，FFN工作节点可以同时发送后向梯度、计算前向传播，并接收后向结果。这种多流执行（图8c）不仅重叠了计算和通信，还最小化了同步开销，实现了接近连续的GPU利用率。

3.  **自适应工作节点分配（Adaptive Worker Allocation）：**
    *   DisagMoE 采用计算-通信屋脊线模型（Roofline Model）来指导 disaggregated 计算和通信资源的自适应分配。
    *   **屋脊线模型：** 扩展了经典的屋脊线模型，以反映AFD架构中注意力 ($I_a$) 和 FFN ($I_f$) 不同的计算-通信比。系统总的GPU数 $W$ 和NICs总数 $M_{tot}$ 在A-Worker（$m$ 个节点，$M_a$ 个NICs）和F-Worker（$n$ 个节点，$M_f$ 个NICs）之间分配。通过调整 $m, n, M_a, M_f$ 的比例，可以动态调整每个组的有效带宽，使FFN组更接近其计算峰值，同时注意力组也保持高效（图9）。
    *   **优化目标：** 通过求解一个混合整数线性规划（MILP）问题（公式7），DisagMoE 首先最小化瓶颈阶段的时间 $T^* = \min \max(T_a, T_f)$，然后在此基础上最大化总算术强度 $(I_{attn} + I_{ffn})$，确保在达到 $T^*$ 的配置中选择最佳的资源利用率。
    *   **精炼算法：** 算法1概述了如何通过屋脊线模型产生初始配置，并结合基于实际剖面数据进行局部搜索，以吸收模型未捕获的影响，从而精炼资源分配。

**实验设置**
*   **实现基础：** DisagMoE 在 Megatron-LM 基础上实现，使用 PyTorch v2.6。代码量为 6K 行 Python 和 2K 行 C++。
*   **硬件条件：** 16个节点，每个节点配备8块NVIDIA H800 GPU（单块80GB显存）、168个CPU。节点间通过ConnectX-7 8x400 GbE NICs互联。节点内GPU通过400GB/s NVLINK互联。
*   **软件版本：** PyTorch v2.6。
*   **负载情况（模型与规模）：**
    *   DeepSeek-MoE (28层, H=2048, E=64, k=4)
    *   GPT-OSS-120B (36层, H=2880, E=128, k=4)
    *   Qwen3-235B-A22B (94层, H=4096, E=128, k=8)
    *   序列长度：4K, 8K, 16K, 32K。
    *   固定局部批次大小（每设备批次大小），因此有效总批次大小随GPU数量线性扩展。
    *   所有基线的EP大小固定为16（跨两个节点）。
*   **对比基线：**
    *   Megatron-LM (1F1B 交错重叠)：NVIDIA的3D并行训练框架。
    *   Tutel：SOTA MoE重叠优化。
    *   Comet：MoE的细粒度计算-通信重叠库。
    *   DualPipe：双向流水线并行重叠前向和后向计算-通信。

**关键对比结果**
*   **端到端性能：** DisagMoE 在所有测试场景（DeepSeek-MoE, GPT-OSS, Qwen3，序列长度4K-32K）下，始终优于所有基线。
    *   相比Megatron-1F1B，DisagMoE 实现了高达 **1.81倍** 的加速。
    *   相比SOTA MoE重叠训练方法（Tutel, Comet, DualPipe），DisagMoE 实现了高达 **1.34倍** 的加速。
    *   具体而言，对Tutel和Comet，加速比通常在1.2-1.5倍。这是因为它们主要依赖操作级重叠，而跨节点通信中，单个FFN操作的计算延迟仅占all-to-all通信总时间的40%-60%，导致大量通信未被隐藏。
    *   对DualPipe，DisagMoE 平均有1.05-1.13倍的加速。DualPipe在热身和冷却阶段仍有大的all-to-all通信气泡，且在稳态下由于注意力与FFN计算-通信比不匹配，仍存在未重叠的尾部。
*   **迭代时间分解：** DisagMoE 实现了更高的计算-通信重叠度。
    *   相比Tutel，非重叠通信时间减少了高达88%。
    *   相比Comet，非重叠通信时间减少了75%。
    *   相比DualPipe，非重叠通信时间减少了45%。
    *   这表明DisagMoE 扩展了有效重叠区域，缩短了端到端迭代时间。
*   **资源分配影响：** 资源分配对性能有显著影响。最佳的注意力与FFN计算资源比率（A:F）随序列长度变化。
    *   在长序列（如16K）下，16:10 的A:F比率性能最佳，比Megatron基线加速1.56倍，比同构配置（16:16）加速1.29倍。
    *   在短序列（如4K）下，由于注意力与FFN计算FLOPs更接近，平衡的16:16或接近的比率表现最佳。
*   **top-k 和 EP 大小影响：** DisagMoE 在不同的top-k和EP设置下，性能均优于基线，加速比在1.08-1.92倍之间。
*   **虚拟阶段影响：** 增加虚拟阶段数量（v）可以提高流水线利用率，减少气泡，直到达到内存限制（v > 16时发生OOM）。这与将面向服务的AFD系统（如HeteroMoE）直接扩展到训练工作负载时遇到的问题一致。

**潜在局限或不足**
1.  **静态形状假设：** DisagMoE 假设预训练工作负载具有稳定的形状（如固定的序列长度和微批次大小）。对于具有动态形状（如强化学习训练）的工作负载，需要在线重新分配资源，这超出了当前工作的范围。
2.  **单一流水线深度：** 当前设计对注意力和FFN组使用相同的流水线深度 $p$。不对称的深度 $(p_A, p_F)$ 是一个有前景的扩展方向，但会引入非平凡的调度复杂性，作为未来工作。
