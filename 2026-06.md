
## IndexMem KVcache驱逐
IndexMem: Learned KV-Cache Eviction with Latent Memory for Long-Context LLM Inference

https://arxiv.org/pdf/2605.25475 港科 浙大 ICML26

1. 提出了一种**可学习的KV-Cache逐出框架**，旨在解决长上下文大型语言模型推理中KV缓存线性增长导致的内存瓶颈和现有启发式逐出策略的信息丢失问题。
2. 核心方法包括一个**可学习的索引器**，用于准确**预测KV token的重要性以进行自适应逐出**，以及一个**轻量级在线更新的latent记忆模块**，通过**残差读取来补偿因逐出而丢失的信息**。
3. Qwen3-8b等单卡小模型，在RULER、Needle-in-a-Haystack和LongBench等基准测试中，相较于现有逐出策略，显著提升了长上下文性能和检索稳定性（在激进逐出下RULER提升高达25点），同时实现了更优的准确性-内存权衡。

**场景与具体问题：**
LLMs 在处理长上下文（如百万 tokens）时展现出卓越能力，但 KV-Cache 的线性增长导致巨大的内存消耗（例如，1M tokens 的 70B 模型需要约 320GB GPU 内存），形成“内存墙”，限制了最大可部署的上下文长度并加剧了解码延迟。现有的 KV-Cache 压缩方法（如量化、低秩近似）虽能减少每 token 内存占用，但无法解决 KV-Cache 随上下文长度线性增长的问题。因此，直接限制内存大小的 KV 驱逐成为关键。

**业界存在哪些不足：**
1.  **启发式驱逐策略的局限性：** 现有的 KV 驱逐策略大多是启发式的，依赖手工设计的代理分数（如 SnapKV 的局部注意力窗口、KeyDiff 的键差异性、Expected Attention 的平均查询统计），难以捕捉 token 重要性的丰富、依赖输入的分布模式。这导致这些方法在预测哪些 token 对未来解码重要时准确性不足，并且通常过度强调局部 token。
2.  **不可逆的信息丢失：** 现有方法永久丢弃被驱逐的 token，导致不可逆的信息丢失。一旦这些 token 在后续变得相关，它们将无法被检索，从而影响模型在长序列上的检索和推理能力。虽然有基于重建的方法（如 KVReviver）尝试缓解，但存在误差放大问题；离线存储方案（如 NOSA、InfLLM）则引入高 CPU-GPU 数据传输延迟，成为吞吐量瓶颈。

**关键观察与假设：**
1.  **可学习的 token 重要性预测：** token 的重要性是依赖于输入的动态属性，传统的启发式规则难以准确捕捉。一个轻量级、可学习的索引器可以更精确地预测 KV token 的重要性，从而实现自适应的 KV 驱逐。
2.  **潜记忆模块缓解信息丢失：** 通过将驱逐的 token 压缩到一个紧凑、固定大小且在线更新的潜记忆模块中，可以保留其信息，并在后续推理中作为残差读取，补偿因驱逐而丢失的注意力贡献，从而减轻不可逆的信息丢失。
3.  **残差补偿优于 token 注入：** 将潜记忆作为额外的 KV token 注入注意力机制可能导致其成为注意力汇聚点（attention sinks）或难以灵活访问。将其作为注意力输出的残差补偿，可以更稳健地集成记忆信息。
4.  **记忆模块的慢权重与快权重分离：** 记忆模块应包含通过梯度训练的“慢权重”以学习通用映射，以及在线更新的“快权重”以适应推理过程中的具体驱逐信息，从而更有效地压缩和检索信息。

**方法核心思路和主要步骤：**
IndexMem 结合了一个可学习的索引器和一个轻量级潜记忆模块。
1.  **可学习索引器 (Learnable Indexer)：**
    *   **作用：** 预测 KV token 的重要性，以决定哪些 token 应被保留，哪些应被驱逐。
    *   **架构：** 采用类似 MQA (Multi-Query Attention) 的设计，轻量化。对于输入序列的隐藏状态 $X \in R^{L \times d_{model}}$ 和预 RoPE (Rotary Positional Embedding) 的查询状态 $Q \in R^{H \times L \times d_{head}}$，索引器输出一个密集的查询到键得分矩阵 $A \in R^{L \times L}$，其中 $A_{s,t}$ 表示位置 $t$ 的键 token 对于位置 $s$ 的查询的重要性。
    *   **重要性计算：** 通过对多头查询进行降维投影得到索引器查询 $q_s = U_q \text{flatten}(Q_s)$，对隐藏状态进行投影得到键 $k_t = U_k X_t$。应用 QK-Norm 归一化。使用学习到的门控向量 $\alpha_s$ 调制索引器头部的贡献，并计算门控相似度 $z_{s,t} = \text{act}(q_s \cdot k_t)$。最终得分 $A_{s,t} = \alpha_s^T z_{s,t} + \text{Mask}_{s,t}$。通过对查询集 $Q$ 进行最大聚合，得到每个 token 的重要性分数 $imp_t = \max_{s \in Q} A_{s,t}$。
    *   **预驱逐能力：** 在 prefill 阶段，索引器可以根据得分预测哪些 KV 条目应被保留，从而仅计算和缓存选定的 KV 状态，避免先计算再驱逐的开销。
    *   **训练：** 冻结骨干 LLM。通过最小化索引器得分与骨干模型注意力得分之间的 KL 散度进行蒸馏训练。损失函数为 $D_{KL}(\text{softmax}(\max_q T(q, \cdot)) || \text{softmax}(\max_q A(q, \cdot)))$，其中 $T$ 是骨干模型的注意力 logits。为节省内存，采用流式（FlashAttention-style）计算方式避免实例化完整的 $L \times L$ 矩阵，并排除 sink tokens 以稳定训练。

2.  **潜记忆模块 (Latent Memory Module)：**
    *   **作用：** 压缩被驱逐 token 的信息到固定大小的记忆状态中，并在需要时提供残差读取以补偿信息丢失。
    *   **残差补偿：** 增强原始注意力输出，即 $o = o_{attn} + g(q) \cdot m(q)$，其中 $o_{attn}$ 仅通过保留的 KV-Cache 计算，而 $m(q)$ 是记忆模块基于当前查询 $q$ 产生的读取， $g(q)$ 是一个门控函数，控制记忆贡献程度。
    *   **结构：** 每个注意力层一个潜记忆模块，跨所有注意力头共享。包含“慢权重” $\theta$（通过训练更新，包括投影层 $\text{Linear}_{\theta}(\cdot)$ 和门控函数 $g(\cdot)$）和“快权重” $(M, b)$（推理时在线更新）。
        *   记忆状态 $M \in R^{d_{mem} \times d_{model}}$，$b \in R^{d_{mem}}$，$d_{mem}$ 为记忆状态维度。
        *   **记忆读取：** $m(q) = \text{Linear}_{\theta}(q)^T M / (\text{Linear}_{\theta}(q)^{\odot 2})^T b + \epsilon$。
        *   **在线写入 (快权重更新)：** 对于驱逐的 KV 对 $(k_i, v_i)$，快权重 $M$ 和 $b$ 通过外积累加和衰减进行更新：
            $M \leftarrow \lambda M + \eta \sum_{i \in E} \text{Linear}_{\theta}(k_i) v_i^T$
            $b \leftarrow \lambda b + \eta \sum_{i \in E} \text{Linear}_{\theta}(k_i)^{\odot 2}$
            其中 $\lambda \in (0, 1]$ 是衰减因子，$\eta > 0$ 是写入强度。
    *   **训练：** 冻结骨干 LLM。通过梯度训练慢权重，使记忆读取与因驱逐而丢失的残差匹配。目标是最小化 MSE 损失 $L_{mem} = \|o - o_{attn} - g(q) \cdot m(q)\|_2^2$，其中 $o$ 是完整注意力输出，$o_{attn}$ 是压缩 KV-Cache 计算的输出。

**实验设置：**
*   **模型：** Qwen3-8B, Mistral-7B-v0.3, Llama-3.1-8B-Instruct。
*   **实现：** 在 KVPRESS (NVIDIA Corporation, 2025) GitHub 仓库中实现 IndexMem 及所有基线，确保驱逐协议一致性。
*   **硬件：** 单块 NVIDIA H800 GPU。
*   **负载情况：** 主要针对长 prefill (prompt 长度 L)，短 decode 的场景。prefill 结束后进行一次性压缩，将 KV-Cache 压缩到 $(1-r)L$。解码期间，每生成 $\tau=128$ 个 token 进行一次周期性压缩，并限制 KV 预算 $B_{max}$。
*   **训练设置：** 冻结骨干 LLM。两阶段训练：1) 单独训练索引器；2) 联合训练索引器和记忆模块。使用 LongAlpaca (Chen et al., 2024) 数据集进行 SFT，采用 chunk-wise KL 目标。学习率调度采用 warmup-stable-decay (WSD) 策略。
    *   索引器超参数：$H_{index} = H/4$, $d_{index} = d_{head}/8$。
    *   记忆模块超参数：$d_{mem} = d_{model}/8$。
*   **对比基线：**
    *   **启发式 KV 驱逐：** ExpectedAttention, KeyDiff, TOVA (Token Omission Via Attention), SnapKV, PyramidKV。
    *   **可学习/表示级 KV 压缩（用于补充比较）：** Locret (Huang et al., 2024), xKV (Chang et al., 2025)。

**关键对比结果：**
1.  **RULER 表现：** 在 RULER-4K 和 RULER-16K 任务上，IndexMem 表现出持续的鲁棒性。在温和压缩 (r ≤ 0.25) 下，IndexMem 几乎与全缓存基线持平。在激进压缩 (r ≥ 0.5) 下，IndexMem 的性能下降更为平缓，尤其在极端预算 (r = 0.75–0.9) 和长上下文 (RULER-16K) 条件下，显著优于启发式方法，证明了学习型索引器的有效性。例如，在 RULER-16K 50% 压缩比下，Qwen3-8B 上的 IndexMem 达到 80.0，远高于 SnapKV 的 62.8。
2.  **Needle-in-a-Haystack (NIAH) 检索：** IndexMem 在 Llama-3.1-8B-Instruct 上表现出最稳定的行为，无论上下文长度 (1K–12.5K) 和“针”的深度 (15%–95%)，都能可靠地检索到“针”。启发式基线（如 PyramidKV 和 SnapKV）则有明显的依赖位置的失效模式，在不利位置性能急剧下降。这表明学习型索引器加上潜记忆模块显著提高了最坏情况下的检索鲁棒性，弥补了索引器可能出现的灾难性遗漏。
3.  **LongBench 结果：** IndexMem 在 LongBench 任务上总体表现最鲁棒，随着压缩比增加，性能下降更平缓。在 `hotpotqa` 和 `multifieldqa en` 等任务上持续领先。在 `triviaqa` 等任务上甚至在适度压缩下略有提升，作者归因于适度驱逐移除了低价值或干扰性 token，提高了上下文的信息密度。
4.  **解码压缩：** 在解码阶段，IndexMem 在受限的 KV 预算 $B_{max}$ 下取得了最佳准确率，表明学习型重要性估计在 KV-Cache 资源受限时特别有益。
5.  **消融研究：** 潜记忆模块的引入对 Indexer-only 方案有持续增益，特别是在激进压缩下。记忆残差显著减少了索引器在极高压缩比下崩溃的灾难性失败（如 `trec` 和 `qasper QA` 任务），验证了存储被驱逐 token 信息的重要性。潜记忆模块与 SnapKV 结合也能带来改进，表明其具有通用性。
6.  **效率分析：** IndexMem 引入的额外参数量极小（Llama-3.1-8B 上索引器 19.92M，记忆模块 0.52M）。在 32K prefill 和 1K decode 的长上下文设置下，indexer-only 版本比全缓存基线更快。添加记忆模块会带来测试时更新开销，但整体延迟仍具竞争力。KV-Cache 内存占用随压缩比的增加而单调下降。

**潜在局限或不足：**
1.  **训练目标和监督信号：** 论文指出当前用于学习 token 重要性和记忆更新的训练目标可能不是最优的，未来可以探索替代目标，以在极端压缩比下进一步提高鲁棒性。
2.  **训练规模：** 目前的训练是在有限的 token 预算和相对适中的规模下进行的，尚未全面评估大规模训练机制或更广泛的模型家族。
3.  **模块化设计：** IndexMem 当前作为骨干模型的附加模块进行训练，骨干模型保持冻结。未来的重要方向是将驱逐和记忆能力原生集成到骨干模型中，通过持续训练实现端到端的联合学习，从而使高效注意力、驱逐决策和记忆读写一体化。
4.  **跨层分数冗余利用：** 论文附录 B 讨论了跨层分数冗余的诊断性研究（如 Running Mean），并连接到 IndexCache-style 的分数重用策略。这表明可以进一步优化索引器计算开销，但该研究本身也指出了一些局限性，例如熵作为置信度代理的不足，以及阈值 $\gamma$ 可能依赖于任务和模型。


## HexiScale 异构GPU训练
HEXISCALE: FACILITATING LARGE LANGUAGE MODEL TRAINING OVER HETEROGENEOUS HARDWARE

https://openreview.net/pdf?id=KgcqSNio0U MLSys26 北大

1. LLM训练通常依赖同构高性能GPU，成本高昂且资源利用率受限；HEXISCALE探索在**异构GPU上训练**，并发现现有同构或部分异构系统在应对不同计算能力、内存和网络带宽时效率低下。
2. 核心贡献在于支持训练计算的**非对称分区**（跨数据、管道和张量模型并行），并提出一种**高效的两阶段分层图分区算法**，将GPU分配和调度问题形式化为受限优化问题进行求解。
3. Llama7B至30B dense模型，HEXISCALE在异构GPU上实现了与同构基线（Megatron, Galvatron）相当的性能，并且比最先进的异构基线（AMP, Espresso, Metis）1.5x～2.4x的吞吐量。

## HEXISCALE: 在异构硬件上促进大型语言模型训练

本文提出了一种名为HEXISCALE的新型系统，旨在解决在异构硬件环境（特别是GPU集群）中高效训练大型语言模型（LLMs）的挑战。传统LLM训练通常依赖于同构的高性能GPU集群，但其高昂的成本限制了LLM训练的普及。HEXISCALE探索了一种替代方案，通过在异构GPU上部署训练计算，以实现更灵活和高效的资源利用。

### 场景与具体问题

当前LLM训练是一个计算密集型任务，通常在具有同构高性能GPU的数据中心进行。然而，这种设置成本高昂，且现有芯片供应商频繁发布新一代AI芯片，导致数据中心往往存在不同代际、不同性能的GPU。这催生了一个重要问题：如何有效利用这些广泛存在但性能各异的异构GPU资源，以更具成本效益和可访问性的方式进行LLM训练。

具体而言，现有最先进的LLM训练系统（如Megatron和DeepSpeed）虽然支持张量模型并行（tensor model parallelism）、流水线并行（pipeline parallelism）和数据并行（data parallelism，包括FSDP），但它们普遍**仅支持同构配置**。这意味着整个训练集群必须在完全对称的设置下运行，要求所有并行组（张量模型并行组、流水线并行组、数据或优化器并行组）具有相同的并行度，并且所有GPU承担相同的计算负载。这种对称性假设导致在异构环境中存在两大根本性挑战：
1.  **不同的GPU计算能力和内存容量：** 异构GPU在计算能力（FLOPS）和内存容量上差异显著。不对称的计算分配会导致性能瓶颈（由最弱GPU决定）和资源利用不足（最强GPU未充分利用）。
2.  **不同的GPU-GPU网络带宽：** 异构的GPU互联（如高速NVLink、PCIe、标准以太网）导致通信时间不均。传统的对称通信模式会导致通信瓶颈，使得快速连接等待慢速连接，从而降低整体效率。

### 业界存在哪些不足

*   **同构系统无法适应异构环境：** Megatron等系统采取完全对称的分区策略，导致在异构环境中GPU利用率低下，因为其并行策略被最低性能的GPU或最慢的网络连接所限制。例如，即使有高性能NVLink连接的GPU，也可能因适应低带宽GPU而被迫采用较低的张量模型并行度，造成资源浪费。
*   **自动并行化算法效率低下：** Alpa和Galvatron等自动并行化算法虽然在同构集群中表现良好，但它们基于对称性假设，并使用高复杂度的算法。在异构环境中，这些算法要么需要过长的执行时间（数小时甚至更长，且难以扩展到数百个GPU），要么在有限的搜索空间中生成次优的并行配置。
*   **现有异构训练系统支持不足：** AMP、Espresso和Metis等系统虽然尝试支持异构训练，但它们在异构感知并行方面的支持有限，限制了算法探索的搜索空间。例如，Metis的深度优先搜索（DFS）调度器倾向于数据并行，除非发生内存不足错误，这可能导致管道气泡（pipeline bubbles）膨胀，因为最佳吞吐量通常需要数据并行和张量模型并行的平衡组合。此外，Metis的DFS算法在大型集群（如240个GPU）上扩展性差，搜索时间过长。

### 关键观察与假设

HEXISCALE的核心观察是：为了在异构环境中高效训练，系统需要能够**不对称地划分计算负载**。这包括：
1.  **不对称的计算负载分配：** 每个数据并行副本（即每个流水线）可以分配不同的批大小，并且可以具有不同的张量模型并行度以及相应不同数量的Transformer层。
2.  **不对称的梯度同步：** 由于不同流水线可能具有不同张量并行度导致的模型参数（及梯度）大小不一，需要定制化的梯度同步机制，将大梯度分块以匹配最小梯度大小，确保通信正确性。
3.  **计算与通信瓶颈的动态变化：** 在异构环境中，数据并行通信和流水线执行时间都可能成为瓶颈。系统需要根据具体情况，自适应地在数据并行带宽和流水线并行带宽之间进行分配优化，以最大化性能。

### 方法核心思路和主要步骤

HEXISCALE的核心思想是实现**完全不对称的并行支持**，并通过一个**两阶段的层次图划分算法**来解决训练计算的分配问题。

**系统设计与实现：**
*   **不对称计算负载划分：**
    *   **流水线内部：** 每个流水线阶段选择一个固定主GPU，以最小化与相邻阶段的通信延迟。该主GPU初始化一个张量模型并行组，并在前向/后向传播中负责激活/梯度在张量并行组内的广播。当相邻流水线阶段的张量模型并行度相同时，HEXISCALE会回退到Megatron的散射/聚集（scatter/gather）通信策略进行集体通信优化。
    *   **流水线间：** 根据GPU的计算能力和内部带宽，将更多的Transformer层和更高的张量并行度分配给高性能GPU组。
    *   **批大小与层分配：** 为了平衡不同流水线的计算速度，可以不对称地分配批大小。同时，层分配会优先将层分配给计算效率高的流水线阶段。
*   **不对称梯度同步：** 通过注册自定义的FSDP通信钩子（communication hooks），即使不同流水线中的模型参数具有不同大小的张量并行度，也能正确同步梯度。它会识别最小梯度大小，并将较大的梯度分块以匹配该大小，确保数据并行通信的正确性。
*   **系统优化：** 支持梯度累积（gradient accumulation）和激活重计算（activation recomputation）以分摊数据并行通信成本和减少内存占用。利用FlashAttention-2/3 API构建Transformer层。集成了定制的基于NCCL的通信层，支持异构GPU间的集体通信。

**调度算法（两阶段层次图划分算法）：**
问题形式化为：在内存限制下，最小化训练迭代时间$T(\sigma) = \text{COMM-COST}(\sigma) + \text{COMP-COST}(\sigma)$。由于该问题是NP-hard，HEXISCALE使用启发式算法寻找近似最优解。

1.  **第一阶段：全局图划分（形成流水线组）**
    *   **目标：** 将GPU设备集合$D$划分为$D_{dp}$个GPU组，每个组形成一个流水线，并最小化数据并行通信的带宽需求。
    *   **步骤：**
        *   **（i）粗化（Coarsen）：** 将所有GPU组织成一个全局图$G=(D, E)$，其中GPU是顶点（权重为其计算能力$c_d$），边是GPU间的通信带宽$\beta_{d,d'}$。使用重边匹配（Heavy Edge Matching, HEM）算法将图粗化为更小的图，通过合并具有高带宽连接的GPU来简化分区问题，使得高带宽的GPU倾向于在同一组内。
        *   **（ii）分区（Partition）：** 对粗化后的图进行分区，使用递归二分法（recursive bisection）将其划分为$D_{dp}$个GPU组$P = \{D_1, D_2, ..., D_{D_{dp}}\}$，以最小化组间切割（CUT）目标函数（即组间通信带宽），同时保持严格的负载均衡。CUT目标函数定义为组间所有边权重的总和。
        *   **（iii）投影与（iv）细化（Project & Refine）：** 将分区结果投影回原始全局图，并使用Kernighan-Lin算法对分区进行细化，以提高分区质量和保持负载均衡。

2.  **第二阶段：流水线内策略确定（形成流水线阶段）**
    *   **目标：** 为每个第一阶段划分出的流水线组$D_i$确定其内部的并行策略（流水线阶段、张量并行度、层分配）和阶段顺序。
    *   **步骤：**
        *   **（i）分组GPU以形成流水线阶段：** 将每个流水线组$D_i$进一步划分为更小的GPU组$P_i = \{D_{i,1}, ..., D_{i,k_i}\}$，这些小组内的GPU具有高带宽连接，将用于构建流水线阶段。这同样采用多级图划分方法。
        *   **（ii）构建流水线阶段：** 对于每个小GPU组$D_{i,k}$，独立搜索其组内最优并行策略（张量并行度、层分配）。这通过模拟不同并行策略的成本模型（见附录B）并选择局部最优解来完成。
        *   **（iii）贪婪搜索流水线阶段顺序：** 将每个组内策略视为一个单独的顶点，构建一个新的图$G'_i$。使用top-τ贪婪算法搜索最优的流水线阶段顺序，以最小化阶段间的通信开销。通过优先连接高带宽的相邻组，有效降低通信成本。

**迭代优化：**
HEXISCALE通过迭代重复两阶段算法来优化并行执行计划，直至达到近似最优解$\sigma^*$。
*   **优化第一阶段：**
    *   枚举不同的流水线数量$D_{dp}$。
    *   自适应地选择网络带宽分配策略：根据历史成本信息，动态选择是最大化（用于数据并行）还是最小化（用于流水线并行）组间带宽，以适应当前瓶颈。
*   **优化第二阶段：**
    *   调整参数$k_i$（每个流水线组内的小GPU组数量），以探索不同的组内策略配置。
*   **批与层分配：** 自动使用贪婪策略分配批大小（按流水线估计效率比例）和层（按计算效率比例）。若遇内存不足，则减少层数并重新分配到内存充足的阶段。
*   **仿真评估：** 每次迭代结束后，使用成本模型（考虑通信带宽矩阵、GPU计算能力、网络延迟）仿真评估生成并行计划的执行成本，并选择成本最低的计划。成本模型能准确预测实际执行成本（偏差小于2.9%）。

### 关键对比结果

1.  **与同构基线比较（相同理论FLOPS的异构GPU vs. 同构GPU）：**

    *   在异构集群中，HEXISCALE表现出强大的竞争力。尽管面临GPU计算不平衡、总内存容量有限和机器间通信频繁等挑战，HEXISCALE的吞吐量仍能达到同构场景下（通过以太网互联的Megatron/Galvatron）的0.83倍至1.01倍。

2.  **与异构基线比较（相同异构集群）：**
    *   HEXISCALE在相同异构集群上比最先进的异构训练系统（AMP, Espresso, Metis）提供了**1.5倍至2.4倍的更高吞吐量**。
    *   **对比Galvatron：** 在异构设置下，Galvatron表现出性能下降，而HEXISCALE实现了**高达2.5倍、平均2.1倍**的吞吐量提升。Galvatron虽灵活，但其并行策略对异构集群仍不最优。
    *   **对比AMP/Espresso：** 在低带宽和高带宽异构互联场景下，HEXISCALE分别实现了**2.29x/2.35x和平均1.75x/1.73x**的更高吞吐量。
    *   **对比Metis：** 在异构设置下，HEXISCALE的吞吐量比Metis高**1.5倍至1.6倍**。Metis的设计（优先数据并行，长流水线配置）导致显著的流水线气泡开销。HEXISCALE通过融合张量模型并行和减少流水线阶段数量来有效缓解此问题。
    *   **对比Cephalo（模拟）：** 在低机器间带宽下，HEXISCALE实现了**高达3.0倍、平均2.8倍**的更高PFLOPS。

3.  **调度算法评估：**
    *   **图划分有效性：** 相较于随机图划分，HEXISCALE的图划分算法在估计吞吐量上收敛到更高的值（LLAMA-2 7B高1.4倍，LLAMA 30B高3.3倍），且在迭代中持续改进。
    *   **可扩展性：** 算法在GPU数量增加时（128到1024个GPU）扩展性良好，1024个GPU的搜索时间少于7分钟，表明调度开销可控。
    *   **与MILP对比：** HEXISCALE的调度算法性能与MILP（混合整数线性规划）方法可比，但搜索时间显著缩短（数分钟 vs. 数小时）。

4.  **系统设计消融研究：**
    *   禁用不对称并行支持：性能平均下降15%，最高下降23%（LLAMA 30B）。这表明不对称并行对于利用不同硬件特性至关重要。
    *   禁用梯度累积：性能平均下降12%，最高下降15%（LLAMA 30B）。这强调了梯度累积在缓解频繁数据并行通信开销中的作用。
    *   延迟分解：HEXISCALE通过减少通信开销和流水线气泡，显著优于Galvatron，特别是在计算不平衡的异构场景中。

### 潜在局限或不足

1.  **仅支持NVIDIA GPU：** 论文提到其实现直接支持NVIDIA GPU上的NCCL，这可能意味着对其他类型GPU（如AMD或Intel）的支持受限或需要额外工作。
2.  **实际部署的复杂性：** 尽管算法效率高，但在真实云环境中，异构集群的动态性（如资源可用性变化、网络拥塞）可能带来额外的管理和调度复杂性，需要更精细的运行时自适应机制。
3.  **成本模型依赖：** 性能预测依赖于准确的成本模型，虽然论文验证了其高精度，但模型参数的获取（如特定GPU间的延迟$\alpha_{d,d'}$和带宽$\beta_{d,d'}$）在实际大规模复杂异构集群中可能仍需繁琐的剖析和更新。
4.  **未开放源代码：** 论文中未提及开源其训练系统，这限制了社区对其方法的直接复现和进一步研究。
   
## TokenPilot 双粒度上下文管理agent
TokenPilot: Cache-Efficient Context Management for LLM Agents

https://arxiv.org/pdf/2606.17016 2026.6.15 浙大 中科大等

代码： https://github.com/zjunlp/LightMem2

中文：https://mp.weixin.qq.com/s/RHFRQkVUcECUYGl_R_KtaA

1. 针对长期运行的LLM Agent场景，研究发现现有的文本修剪与动态内存回收策略会频繁触发序列布局改变，进而**导致严重的Prefix Mismatch和Cache Invalidation问题**，大幅增加了推理成本。
2. 提出了**TokenPilot双粒度上下文管理agent**，通过全局**的Ingestion-Aware Compaction标准化输入**布局以保障Prompt缓存的物理连续性，并结合**本地的Lifecycle-Aware Eviction**执行保守的批处理淘汰策略。
3. TokenPilot在PinchBench和Claw-Eval基准测试中，在保持任务性能的同时，显著降低了推理成本，其中在连续模式下最高可实现87%的成本削减。

### 1. 场景与问题
随着 LLM 代理从对话助手转向状态执行控制器（Stateful execution controllers），连续的多轮交互导致执行轨迹（Execution traces）迅速膨胀。这使得推理成本急剧上升，且由于现有方法在压缩文本时频繁改动 prompt 布局，导致后端 KV cache 发生大规模失效（Cache invalidation），带来了严重的预填充（Pre-fill）惩罚。

### 2. 业界不足
现有的上下文管理方案主要关注文本层面的压缩（如 pruning 或 summarization），即通过移除低效 Token 来缩小体积。然而，这些方法往往忽略了底层硬件的 **Prompt Cache 连续性**，通过不可控的序列突变（Sequence mutations）破坏了缓存的命中布局，导致“省下了文本长度，却增加了预填充成本”的现象。

### 3. 关键观察与假设
*   **核心矛盾**：存在文本稀疏性（Text sparsity）与硬件缓存对齐（Prompt cache continuity）之间的 trade-off。
*   **假设**：高效的系统必须在“观测输入阶段”保护物理前缀连续性，并在“结构化内存淘汰阶段”根据轨迹的剩余效用（Residual utility）延迟过期处理，以实现文本减少与硬件缓存对齐的协同。

### 4. 方法核心思路与步骤
TokenPilot 采用双粒度管理架构：

*   **全局：Ingestion-Aware Compaction（摄入感知压缩）**
    *   **Prefix Stabilization（前缀稳定）**：通过 canonicalization 操作，将 runtime 变量（如路径、时间戳）替换为静态占位符，确保从第一轮起即获得 byte-identical 的 prompt 前缀，从而规避缓存不命中。
    *   **Observation Reduction（观测压缩）**：区分内部指令消息 ($\Omega_{int}$) 与外部环境反馈 ($\Omega_{env}$)，对环境反馈进行确定性的结构化预览（Structured preview）压缩，并建立 Artifact registry 注册表，当代理需深度分析时通过 Recovery tool 自动回调原始数据，确保操作安全性。
    *   形式化描述：$\hat{U}(m) = \mathbb{1}[m \in \Omega_{int}] + \max(\gamma(m), G(m))\mathbb{1}[m \in \Omega_{env}]$，其中 $G(m)$ 为 ingestion gate。

*   **局部：Lifecycle-Aware Eviction（生命周期感知淘汰）**
    *   **保守的淘汰策略**：将上下文片段的状态划分为 $\{active, completed, evictable\}$。仅在任务残留价值 $\Psi_j = \emptyset$ 且完成任务后，才将片段标记为可淘汰。
    *   **批量触发**：为了抑制状态震荡，引入 batch-turn 调度（以 $B$ 轮为单位），通过 estimator 进行保守的在线评估，避免频繁的页表换出。

### 5. 实验设置
*   **基准测试**：在 **PinchBench** 和 **Claw-Eval** 数据集上进行测试，对比了 LLMLingua-2, SelectiveContext, MemOS 等 10 种基线方法。
*   **模式**：Isolated（任务边界重置）与 Continuous（历史持续存在）。
*   **技术栈**：Agent backbone 为 GPT-5.4-mini；状态 estimator 使用 Qwen3.5-35B-A3B；基于 LightMem2 实现。

### 6. 关键对比结果
*   **经济性**：在持续模式（Continuous mode）下，PinchBench 和 Claw-Eval 的推理成本分别降低了 **61%** 和 **87%**。
*   **缓存效率**：通过前缀稳定化，PinchBench 的 macro cache hit rate 从 38.7% 提升至 79.2%；Claw-Eval 从 67.2% 提升至 83.1%。
*   **任务性能**：在显著降低成本的同时，TokenPilot 保持了与 Vanilla 相当甚至更优的各项指标（Prod, Res, Write, Code 等）。

### 7. 潜在局限与不足
*   **依赖特定能力**：依赖后端 KV prompt caching 技术支持，若供应商不支持该特性则无效。
*   **估计器误差**：在高度模糊或稀疏的交互模式下，基于模型的状态 estimator 存在错判风险。
*   **参数敏感性**：阈值 $\tau$ 和批处理大小 $B$ 需要根据特定部署环境进行微调。
*   **任务混合度**：对于高度杂乱、异构的任务流，工具定义模式的频繁突变仍可能导致 prefix reuse 性能下降。

#### 1. 场景与问题
Agentic Kernel Optimization（智能体内核优化）旨在利用大语言模型（LLMs）自动化GPU内核调优，通过迭代生成、验证和性能分析（Profiling）来探索高性能内核空间。然而，该过程在系统层面存在三大效率瓶颈：
*   **(C1) 生成延迟高**：LLM推理占据了迭代时间总量的70%-99%，严重限制了单位时间内的迭代次数。
*   **(C2) 分析反馈不足**：由于大量生成的内核未能通过验证，导致59%-64%的迭代无法产生有效的性能分析数据，无法驱动后续优化。
*   **(C3) 资源利用率低下**：验证和分析GPU在生成阶段处于空闲状态，利用率仅为4.2%-17.6%。

#### 2. 关键观察与核心假设
研究发现，LLM的推理过程并非瞬时完成，其推理轨迹中暴露了一个**时间窗口**，可以在推理结束前通过对推理前缀（Reasoning Prefix）的条件化，提前分叉产生非推理生成的候选内核。
*   **假设**：利用推理中间状态作为提示（Prompt），可以在推理未完成时生成高质量内核，并能在检测到“满足阈值的内核”时提前终止推理生成，从而减少延迟并提升资源利用率。

#### 3. 方法核心思路：SpecGen
SpecGen 是一个集成了 **SpecController** 和 **ElasticScheduler** 的内核优化系统，无需修改底层的LLM或搜索算法。

*   **SpecController（ speculative generation）**：
    *   **触发器监控**：利用解析器从推理轨迹中捕捉“内核设计决策”（如瓦片形状、代码块补全），触发 $K$ 个非推理生成分叉。
    *   **提前终止**：一旦某个候选内核的性能超过了历史性能均值 $\tau = \text{mean}(H)$，**系统立即终止主推理任务，以此规避长尾生成耗时**。// 这合理么？
*   **ElasticScheduler（资源与调度）**：
    *   **弹性池分配**：基于上一轮的队列压力，动态重新划分验证和分析GPU池。
    *   **优先级队列**：验证队列采用后到先服务（LAF）策略以处理最新的设计优化，分析队列采用FIFO策略以减少反馈延迟。
    *   **远程KV缓存**：利用验证/分析GPU闲置的内存作为远程KV缓存存储，通过RDMA技术（Mooncake）消除重复计算推理前缀的开销。

#### 4. 实验设置
*   **实现基础**：在 KernelBench 框架之上实现。
*   **硬件平台**：H200 GPU集群，支持NVLink和RoCEv2。
*   **软件环境**：PyTorch 2.10, CUDA 12.9, NCU 2025.1.0, Ninja 1.13.0。
*   **模型**：GLM-5.1 和 DeepSeek-V4-Pro。
*   **负载**：涵盖 KernelBench Level 1/2/3 的20个典型算子任务。
*   **对比基线**：CudaForge, AlphaEvolve, KernelAgent。

#### 5. 关键对比结果
*   **性能效率**：SpecGen 将端到端执行时间缩短了 1.68–1.82 倍。
*   **分析反馈**：Profiling反馈数量提升了 1.58–1.98 倍。
*   **资源利用率**：验证/分析GPU利用率从 4.2%–17.6% 大幅提升至 88.2%–96.1%。
*   **内核性能**：在相同的时间预算下，最终内核的加速比（Speedup）较基线提升了 1.24–1.91 倍。

#### 6. 潜在局限与不足
*   **触发器噪声**：系统依赖于正则表达式解析推理轨迹，对于极其复杂或非标准格式的推理输出，触发准确性可能波动。
*   **终止阈值敏感性**：虽然均值阈值 $\tau$ 表现稳健，但在算子极端不稳定的优化初期，触发提前终止的决策仍存在一定的随机性，可能导致过早舍弃潜在的最优解，虽有回溯机制但增加了复杂度。
*   **资源依赖**：该系统依赖于闲置显存作为远程KV缓存，在极度缺乏GPU资源的极端小规模实验环境下，缓存收益会受限于显存容量。
## AutoMegaKernel
AutoMegaKernel: A Statically-Checked Agent Harness for Self-Retargeting Megakernel Synthesis

https://arxiv.org/pdf/2606.09682 2026.6.8

https://github.com/RightNow-AI/AutoMegaKernel

#### 1. 场景与具体问题
在单流（Single-stream）大语言模型（LLM）的 $Batch-1$ 推理中，算术强度接近 1，导致推理过程受到显存带宽（HBM Bandwidth）的严格限制，即延迟下限 $t_{min} = \text{weight\_bytes} / \text{HBM\_bandwidth}$。目前主流的 PyTorch 或 cuBLAS 方案因频繁的算子内核启动（Kernel Launch）和算子间激活值（Activations）在 HBM 中的往返拷贝，产生了巨大的启动开销和带宽浪费，无法达到理论性能下限。

#### 2. 业界现状与不足
*   **现有方案（如 vLLM, SGLang）**：主要优化高吞吐场景，而非针对 $Batch-1$ 低延迟。
*   **Mirage Persistent Kernel (MPK)**：虽实现了持久内核，但**缺乏静态死锁/竞态检查机制**，且难以供编码智能体进行安全修改。
*   **手工构建的 Megakernel**：针对特定模型开发，缺乏通用性与可移植性。

#### 3. 关键观察与假设
*   **核心观察**：LLM 前向传播在本质上是一个有向无环图（DAG），通过单次启动（Single-launch）的持久化协作内核可以消除算子边界，实现全模型融合。
*   **关键假设**：若能通过一套冻结的验证逻辑（Static Schedule Validator）对算子依赖图进行静态认证，即可确保该调度在运行时的死锁/竞态免疫，从而将“正确性”转化为编译时的一个结构性不变量。

#### 4. 方法核心思路与步骤
AMK 采用四层架构，核心是通过一个“**自动搜索与静态验证**”的闭环完成 megakernel 合成：
*   **Layer 0 (VM)**：手写的、架构相关的**持久化协作 VM**，**每SM驻留一个线程块**，利用单调计数器（Monotonic Counters）实现**细粒度的跨任务同步**。
*   **Layer 1 (Instructions)**：ABI 兼容的微内核集合（如 GEMV, Attention, RMSNorm 等）。
*   **Layer 2 (Scheduler)**：将 HuggingFace 模型下沉为 Tiled Task-DAG。核心在于 `validate()` 函数：通过 Kahn 算法检测循环依赖，检查等待阈值 $1 \le t \le \#producers$，并在静态阶段通过 bitmask 遍历验证所有读写流的合法性。
*   **Loop (Self-Improvement)**：智能体驱动的自动搜索循环。流程为：`propose`（提出调度配置）$\rightarrow$ `validate`（静态门控）$\rightarrow$ `correctness-check`（对比 CPU Reference VM）$\rightarrow$ `measure`（基于 drift-robust 的测量）$\rightarrow$ `keep/revert`。

#### 5. 实验设置
*   **硬件条件**：覆盖从消费级（RTX 5090）到数据中心推理级（L4, L40S, A10G）及训练级（A100, H100）的 7 种 GPU。
*   **软件版本**：`torch 2.11.0+cu128`，CUDA 12.8/13.1。
*   **负载与基线**：以 $Batch-1$ 的 decode position 0 为基准（权重受限场景）。对比基线包括 cuBLAS（CUDA-graph 模式）及 vLLM 等。
*   **正确性验证**：在 7,160 个对抗性调度样本上测试验证器，**保持 0 误报率**（False-accepts）。

#### 6. 关键对比结果
*   **正确性**：生成的 megakernel 在 10/10 个测试模型上**与 HuggingFace eager 模式实现 token-for-token 一致**，且在 SmolLM2-135M 上困惑度（Perplexity）误差仅 $2.5 \times 10^{-7}$。
*   **性能 win**：在推理级 GPU（L4, L40S, A10G）上，AMK 自动搜索生成的 **int8 weight-only (W8A16)** megakernel 在 $Batch-1$ 场景下击败了 CUDA-graphed cuBLAS bf16，其中 NVIDIA L4 上性能提升最高可达 1.33×。
*   **性能局限**：在训练级 GPU（A100/H100）上，由于跨 SM 的同步开销（Cross-SM sync）占比较大，AMK 略逊于 cuBLAS。bf16 等精度对比下，AMK 整体性能比 cuBLAS 慢约 13%。

#### 7. 潜在局限与不足
*   **Kernel Quality**：当前 GEMV 实现尚未完全发挥底层架构潜力，在等精度 bf16 任务上仍存在 13% 的性能落差。
*   **同步开销**：随着 HBM 带宽提升，Per-tile的** grid-wide 同步变成了性能瓶颈**，需要进一步研发粗粒度同步调度。
*   **资源限制**：由于缺乏大规模计算资源，目前未能进行深度 tensor-core 调优，未来需进一步优化算子实现。
*   **架构限制**：仅**支持 Llama 系列的特定结构**，对于 MoE、复杂 RoPE 缩放等变体存在支持盲点（仅支持 config-level 检查）。

## StreamIndex
StreamIndex: Memory-Bounded Compressed Sparse Attention via Streaming Top-k

paper: https://arxiv.org/pdf/2605.02568 2026.5.4 RightNow

code: https://github.com/RightNow-AI/StreamIndex

1. 针对 DeepSeek-V4 等大模型在长上下文（1M token）任务中面临的 Compressed Sparse Attention (CSA) **索引计算的内存瓶颈**，研究发现现有的公共实现因会预先物化**巨大的FP32中间得分张量**，导致在单卡 H200 上出现内存溢出（OOM）。
2. 提出了 STREAMINDEX，通过将索引计算重构为基于 Triton 的分块（chunked）partition-merge 驱动程序，利用流式 Top-k 不变性成功消除了对完整中间张量的物化需求，并将 mask 内存优化至与序列长度无关的 $O(c_S \cdot c_T)$。
3. H200单卡，v4-flash，对比官方，在保持与基准模型位级精确（bit-exact）的一致性前提下，将 CSA 的可处理序列长度从 65K 扩展至 1M，实现了 32 倍的内存效率提升，并能与现有的 TileLang 注意力内核无缝集成。

#### 1. 场景与具体问题
DeepSeek-V3.2 及 V4 模型引入了**压缩稀疏注意力 (Compressed Sparse Attention, CSA)** 机制，通过 Lightning Indexer 对压缩后的 Key 进行打分，并根据每条 Query 选择 Top-k 个 Key 进行稀疏注意力计算。在 V4-Flash 架构下（Head Indexer $HI = 64$，压缩率 $m = 4$），该机制的核心瓶颈在于 Indexer 步骤会产生一个形状为 $[B, S, HI, T]$ 的 FP32 中间评分tensor（其中 $T=S/m$）。在序列长度 $S=65,536$ 时，**该tensor高达 256 GB**，远超单卡 NVIDIA H200 (140 GB HBM) 的内存容量，导致在长文本推理的注意力计算前即发生内存溢出 (OOM)。

#### 2. 业界存在的不足
现有的长文本优化技术（如 FlashAttention, PagedAttention, Ring Attention）主要关注注意力计算本身（softmax 与矩阵乘法）或 KV Cache 的管理，未能解决 CSA 架构中 Indexer 阶段产生的海量中间评分缓冲区问题。目前的实现方式通常采取全量物化 (Materialize) 评分矩阵的策略，限制了模型在超长上下文（如 1M token）下的部署。

#### 3. 关键观察与假设
*   **Indexer 的可分离性**：Lightning Indexer 的评分函数 $I(t, s) = \sum_{h=1}^{HI} w_{t,h} \cdot \text{ReLU}(q_{t,h} \cdot K_s)$ 是 Key 轴可分离的，且不存在跨 Key 的归一化混合，这使得 Top-k 选择满足“分治”特性。
*   **瓶颈定位**：CSA 的扩展性受限于 Indexer 阶段的内存峰值，而非注意力 Kernel 本身的算力。

#### 4. 方法核心思路与主要步骤
作者提出了 **STREAMINDEX**，这是一种基于 Triton 实现的流式 Top-k 驱动器。
*   **分块分区-合并 (Chunked Partition-Merge)**：不一次性计算完整的评分张量，而是将 Query 和 Key 划分为 $[c_S, c_T]$ 大小的 Tile。
*   **算子融合**：使用融合的 Triton Kernel 计算小块分数，在内部完成 Head 求和，避免物化 $[B, S, HI, T]$ 中间结果。
*   **流式处理**：通过嵌套循环，对每个 Tile 计算 Top-k，并通过合并（Merge）操作更新全局 Top-k 缓冲区。峰值 HBM 从 $O(B \cdot S \cdot HI \cdot T)$ 降至 $O(B \cdot c_S \cdot c_T)$。
*   **按需 Masking**：引入每块因果掩码 (Per-chunk causal-mask)，将掩码内存需求从 $O(S \cdot T)$ 降低到 $O(c_S \cdot c_T)$，支撑 1M 序列长度扩展。

#### 5. 实验设置
*   **硬件条件**：单卡 NVIDIA H200 (140 GB HBM3e)。
*   **软件版本**：CUDA 13, Triton 3.7, PyTorch 2.13 nightly。
*   **负载情况**：基于合成但符合 V4-Flash 统计特征的分布（RoPE, Hadamard 旋转等）进行验证。
*   **对比基线**：与 V4-Flash 官方参考实现（全量物化法）及 TileLang 提供的注意力 Kernel 进行对比。

#### 6. 关键对比结果
*   **量级扩展**：在 V4-Flash 维度下，原实现 $S=65,536$ 时 OOM；STREAMINDEX 在 $S=1,048,576$ 时仍能运行，仅占用 6.21 GB 峰值 HBM，实现了 32 倍的长度扩展。
*   **性能表现**：在 $S=262,144$ 的全流水线测试中，原实现 OOM，STREAMINDEX 配合 TileLang 注意力 Kernel 可在 1.97 秒内完成计算。
*   **准确性**：通过多组设计空间扫描，验证了流式 Top-k 与全量物化法在 Set-overlap Recall 上实现了比特级 (Bit-exact) 的完全一致。

#### 7. 潜在局限或不足
*   **缺乏真实权重验证**：由于单卡内存限制，未能加载 270 GB 规模的 V4-Flash 完整权重进行端到端推理任务（如 Needle-in-a-haystack）。
*   **输入重读**：为了实现极致的低显存占用，STREAMINDEX 在分块处理过程中会对输入进行多次读取，在小序列长度下比全量物化实现慢（但解决了 OOM 问题）。
*   **排序确定性**：依赖 `torch.topk` 的 Tie-breaking 规则，未在代码层面强制定义词典序的确定性比较器，但已通过实验证明其 Set 集合 parity 满足需求。

#### 1. 场景与具体问题
MoE (Mixture-of-Experts) 模型通过在推理时激活少量的专家（Experts）参数来平衡模型容量与计算成本。核心瓶颈在于融合后的 MoE Kernel（Fused MoE Kernel），其性能深受批大小（Batch size）及运行时专家路由分布（Expert routing distribution）的影响。在生产环境中，路由往往呈现高度偏斜（Skewed），而现有的调度系统仅依据 Batch size 进行固定配置，导致 10%–70% 的内核吞吐量未被有效利用。

#### 2. 业界存在的不足
目前主流的生产推理系统（如 vLLM, SGLang, Alpha-MoE, DeepGEMM, FlashInfer）均采取“静态调度（Static Dispatch）”策略。这些系统在路由分布恒定为均匀分布（$\beta=1.0$）的假设下进行优化，忽略了输入依赖的路由波动。此外，通过增加调优维度来覆盖更多场景会导致搜索空间呈指数级增长，使得自动调优（Autotuning）成本高昂且不切实际。

#### 3. 关键观察与假设
*   **路由偏斜**：实际生产中，每层仅有 8–14% 的专家被激活，路由平衡度 $\beta \approx 0.5$（$\beta = H(c)/ \ln E$），这与静态调度假设的均匀分布严重偏离。
*   **物理瓶颈**：内核性能受限于计算密度（$\rho$）、L2 缓存压力（$\lambda$）、波前利用率（$\omega$）和 K-维度归约深度（$\kappa$）。
*   **波前量化（Wave Quantization）**：CTA（Cooperative Thread Array）网格在不同路由分布下会导致 GPU SM 出现波前碎片，静态调度无法捕捉这种非线性的性能跳变。

#### 4. 核心方法：RaMP
RaMP 采用“路由感知（Routing-aware）”的动态调度框架，包含三个核心组件：
*   **性能区域分析（Performance-Region Analysis）**：利用四个几何参数（$\rho, \lambda, \kappa, \omega$）从硬件常量中预测各优化策略（如 GROUP M swizzle, Split-K）的适用性。
*   **四参数波前成本模型（Wave Cost Model）**：将执行时间建模为四个具有物理意义的项：
    $T(c) = a(c) + b(c) \frac{g}{SM} + c(c) g + d(c) \log(g+1)$
    其中 $g$ 为 CTA 网格大小，$a$ 为启动开销，$b$ 为波前调度成本，$c$ 为 HBM 内存访问成本，$d$ 为子波前（Sub-wave）阶段的边际利用率收益。
*   **多态内核（Polymorphic Kernel）**：基于 CuTe DSL 开发的融合内核，暴露 134–268 种配置，通过 JIT 编译及运行时成本评估实现秒级选优。

#### 5. 实验设置
*   **硬件条件**：NVIDIA H200 SXM (132 SMs)。
*   **软件栈**：CUDA 12.8, PyTorch 2.9, vLLM v0.9, CUTLASS 4.0。
*   **负载情况**：涵盖 OLMoE, Qwen3, DSv3, Mixtral 等 8 种架构（其中 3 种为未见过模型），使用 MMLU, GSM8K 及真实聊天语料进行压力测试。
*   **对比基线**：Triton FP8 (vLLM 默认), DeepGEMM, FlashInfer CUTLASS 以及 Alpha-MoE 的 JIT 调度。

#### 6. 关键对比结果
*   **预测精度**：在 8 种架构上，成本模型对比穷举搜索的平均遗憾（Mean regret）仅为 0.93%。
*   **内核加速**：在真实路由偏斜（$\beta=0.5$）下，相较于静态调度获得 1.22× 的几何平均加速；对 Alpha-MoE 内核应用 RaMP 调度可提升 1.14×。
*   **端到端性能**：在 vLLM 服务中，RaMP 在 OLMoE 上相较于 Triton FP8 实现了 1.30× 的 TPOT 加速，相较于 DeepGEMM 提升 1.41×，相较于 FlashInfer 提升 1.13×。
*   **泛化能力**：通过 10–24 分钟的一次性分析，无需任何模型特有调优即可在未知架构上达到最优性能。

#### 7. 潜在局限
*   **硬件依赖**：当前模型基于 Hopper (SM 90a) 架构设计，若迁移至 Blackwell 架构需重新进行少量的配置分析。
*   **通信开销**：该方案侧重于单 GPU 内的计算内核调度，对于多卡环境下的专家并行通信（Expert-parallel communication）开销未纳入优化范围。
*   **极端场景**：在极小批（$S=1$）或极度均匀的路由场景下，动态调度带来的收益会收敛于静态调度。
