
## StreamIndex
StreamIndex: Memory-Bounded Compressed Sparse Attention via Streaming Top-k

paper: https://arxiv.org/pdf/2605.02568 2026.5.4 RightNow

code: https://github.com/RightNow-AI/StreamIndex

1. 针对 DeepSeek-V4 等大模型在长上下文（1M token）任务中面临的 Compressed Sparse Attention (CSA) **索引计算的内存瓶颈**，研究发现现有的公共实现因会预先物化**巨大的FP32中间得分张量**，导致在单卡 H200 上出现内存溢出（OOM）。
2. 提出了 STREAMINDEX，通过将索引计算重构为基于 Triton 的分块（chunked）partition-merge 驱动程序，利用流式 Top-k 不变性成功消除了对完整中间张量的物化需求，并将 mask 内存优化至与序列长度无关的 $O(c_S \cdot c_T)$。
3. H200单卡，v4-flash，对比官方，在保持与基准模型位级精确（bit-exact）的一致性前提下，将 CSA 的可处理序列长度从 65K 扩展至 1M，实现了 32 倍的内存效率提升，并能与现有的 TileLang 注意力内核无缝集成。
PyTorch ≥ 2.10, Triton ≥ 3.5, CUDA 12.x or 13.x, Python ≥ 3.11.

#### 1. 场景与具体问题
DeepSeek-V3.2 及 V4 模型引入了**压缩稀疏注意力 (Compressed Sparse Attention, CSA)** 机制，通过 Lightning Indexer 对压缩后的 Key 进行打分，并根据每条 Query 选择 Top-k 个 Key 进行稀疏注意力计算。在 V4-Flash 架构下（Head Indexer $HI = 64$，压缩率 $m = 4$），该机制的核心瓶颈在于 Indexer 步骤会产生一个形状为 $[B, S, HI, T]$ 的 FP32 中间评分tensor（其中 $T=S/m$）。在序列长度 $S=65,536$ 时，**该tensor高达 256 GB**，远超单卡 NVIDIA H200 (140 GB HBM) 的内存容量，导致在长文本推理的注意力计算前即发生内存溢出 (OOM)。

#### 2. 业界存在的不足
现有的长文本优化技术（如 FlashAttention, PagedAttention, Ring Attention）主要关注注意力计算本身（softmax 与矩阵乘法）或 KV Cache 的管理，未能解决 CSA 架构中 Indexer 阶段产生的海量中间评分缓冲区问题。目前的实现方式通常采取全量物化 (Materialize) 评分矩阵的策略，限制了模型在超长上下文（如 1M token）下的部署。

#### 3. 关键观察与假设
*   **Indexer 的可分离性**：Lightning Indexer 的评分函数 $I(t, s) = \sum_{h=1}^{HI} w_{t,h} \cdot \text{ReLU}(q_{t,h} \cdot K_s)$ 是 Key 轴可分离的，且不存在跨 Key 的归一化混合，这使得 Top-k 选择满足“分治”特性。
*   **瓶颈定位**：CSA 的扩展性受限于 Indexer 阶段的内存峰值，而非注意力 Kernel 本身的算力。

#### 4. 方法核心思路与主要步骤
作者提出了 **STREAMINDEX**，这是一种基于 Triton 实现的流式 Top-k 驱动器。
*   **分块分区-合并 (Chunked Partition-Merge)**：不一次性计算完整的评分张量，而是将 Query 和 Key 划分为 $[c_S, c_T]$ 大小的 Tile。
*   **算子融合**：使用融合的 Triton Kernel 计算小块分数，在内部完成 Head 求和，避免物化 $[B, S, HI, T]$ 中间结果。
*   **流式处理**：通过嵌套循环，对每个 Tile 计算 Top-k，并通过合并（Merge）操作更新全局 Top-k 缓冲区。峰值 HBM 从 $O(B \cdot S \cdot HI \cdot T)$ 降至 $O(B \cdot c_S \cdot c_T)$。
*   **按需 Masking**：引入每块因果掩码 (Per-chunk causal-mask)，将掩码内存需求从 $O(S \cdot T)$ 降低到 $O(c_S \cdot c_T)$，支撑 1M 序列长度扩展。

#### 5. 实验设置
*   **硬件条件**：单卡 NVIDIA H200 (140 GB HBM3e)。
*   **软件版本**：CUDA 13, Triton 3.7, PyTorch 2.13 nightly。
*   **负载情况**：基于合成但符合 V4-Flash 统计特征的分布（RoPE, Hadamard 旋转等）进行验证。
*   **对比基线**：与 V4-Flash 官方参考实现（全量物化法）及 TileLang 提供的注意力 Kernel 进行对比。

#### 6. 关键对比结果
*   **量级扩展**：在 V4-Flash 维度下，原实现 $S=65,536$ 时 OOM；STREAMINDEX 在 $S=1,048,576$ 时仍能运行，仅占用 6.21 GB 峰值 HBM，实现了 32 倍的长度扩展。
*   **性能表现**：在 $S=262,144$ 的全流水线测试中，原实现 OOM，STREAMINDEX 配合 TileLang 注意力 Kernel 可在 1.97 秒内完成计算。
*   **准确性**：通过多组设计空间扫描，验证了流式 Top-k 与全量物化法在 Set-overlap Recall 上实现了比特级 (Bit-exact) 的完全一致。

#### 7. 潜在局限或不足
*   **缺乏真实权重验证**：由于单卡内存限制，未能加载 270 GB 规模的 V4-Flash 完整权重进行端到端推理任务（如 Needle-in-a-haystack）。
*   **输入重读**：为了实现极致的低显存占用，STREAMINDEX 在分块处理过程中会对输入进行多次读取，在小序列长度下比全量物化实现慢（但解决了 OOM 问题）。
*   **排序确定性**：依赖 `torch.topk` 的 Tie-breaking 规则，未在代码层面强制定义词典序的确定性比较器，但已通过实验证明其 Set 集合 parity 满足需求。
  
## RaMP FusedMoE
RaMP: Runtime-Aware Megakernel Polymorphism for Mixture-of-Experts

https://arxiv.org/pdf/2604.26039 2026.4.28 Hippocratic AI

1. 针对MoE推理中因静态kernel选择调度 忽略路由分布导致的性能瓶颈，研究发现实际生产中 MoE专家路由高度倾斜，仅依赖 Batch Size 进行调度会损失 10–70% 的内核吞吐量。
2. Routing-aware: 引入性能区域分析和基于硬件常数的四参数波浪成本模型，运行时根据专家直方图动态选择最优内核配置，并实现了与内核实现无关的通用适配。
3. H200，vLLM 0.9，相较于 Triton、DeepGEMM 和 FlashInfer，在 OLMoE, DeepSeekV3, Qwen3等模型kernel实现upto 1.7x加速 (Deepseek-v3)，同时保持了极低的运行时调度开销。
cuda12.8; cutlass 4.0； 1,738-line CuTe DSL fused 3-phase kernel (W8A8 FP8 128×128 block scales) 合计134–268 config

#### 1. 场景与具体问题
MoE (Mixture-of-Experts) 模型通过在推理时激活少量的专家（Experts）参数来平衡模型容量与计算成本。核心瓶颈在于融合后的 MoE Kernel（Fused MoE Kernel），其性能深受批大小（Batch size）及运行时专家路由分布（Expert routing distribution）的影响。在生产环境中，路由往往呈现高度偏斜（Skewed），而现有的调度系统仅依据 Batch size 进行固定配置，导致 10%–70% 的内核吞吐量未被有效利用。

#### 2. 业界存在的不足
目前主流的生产推理系统（如 vLLM, SGLang, Alpha-MoE, DeepGEMM, FlashInfer）均采取“静态调度（Static Dispatch）”策略。这些系统在路由分布恒定为均匀分布（$\beta=1.0$）的假设下进行优化，忽略了输入依赖的路由波动。此外，通过增加调优维度来覆盖更多场景会导致搜索空间呈指数级增长，使得自动调优（Autotuning）成本高昂且不切实际。

#### 3. 关键观察与假设
*   **路由偏斜**：实际生产中，每层仅有 8–14% 的专家被激活，路由平衡度 $\beta \approx 0.5$（$\beta = H(c)/ \ln E$），这与静态调度假设的均匀分布严重偏离。
*   **物理瓶颈**：内核性能受限于计算密度（$\rho$）、L2 缓存压力（$\lambda$）、波前利用率（$\omega$）和 K-维度归约深度（$\kappa$）。
*   **波前量化（Wave Quantization）**：CTA（Cooperative Thread Array）网格在不同路由分布下会导致 GPU SM 出现波前碎片，静态调度无法捕捉这种非线性的性能跳变。

#### 4. 核心方法：RaMP
RaMP 采用“路由感知（Routing-aware）”的动态调度框架，包含三个核心组件：
*   **性能区域分析（Performance-Region Analysis）**：利用四个几何参数（$\rho, \lambda, \kappa, \omega$）从硬件常量中预测各优化策略（如 GROUP M swizzle, Split-K）的适用性。
*   **四参数波前成本模型（Wave Cost Model）**：将执行时间建模为四个具有物理意义的项：
    $T(c) = a(c) + b(c) \frac{g}{SM} + c(c) g + d(c) \log(g+1)$
    其中 $g$ 为 CTA 网格大小，$a$ 为启动开销，$b$ 为波前调度成本，$c$ 为 HBM 内存访问成本，$d$ 为子波前（Sub-wave）阶段的边际利用率收益。
*   **多态内核（Polymorphic Kernel）**：基于 CuTe DSL 开发的融合内核，暴露 134–268 种配置，通过 JIT 编译及运行时成本评估实现秒级选优。

#### 5. 实验设置
*   **硬件条件**：NVIDIA H200 SXM (132 SMs)。
*   **软件栈**：CUDA 12.8, PyTorch 2.9, vLLM v0.9, CUTLASS 4.0。
*   **负载情况**：涵盖 OLMoE, Qwen3, DSv3, Mixtral 等 8 种架构（其中 3 种为未见过模型），使用 MMLU, GSM8K 及真实聊天语料进行压力测试。
*   **对比基线**：Triton FP8 (vLLM 默认), DeepGEMM, FlashInfer CUTLASS 以及 Alpha-MoE 的 JIT 调度。

#### 6. 关键对比结果
*   **预测精度**：在 8 种架构上，成本模型对比穷举搜索的平均遗憾（Mean regret）仅为 0.93%。
*   **内核加速**：在真实路由偏斜（$\beta=0.5$）下，相较于静态调度获得 1.22× 的几何平均加速；对 Alpha-MoE 内核应用 RaMP 调度可提升 1.14×。
*   **端到端性能**：在 vLLM 服务中，RaMP 在 OLMoE 上相较于 Triton FP8 实现了 1.30× 的 TPOT 加速，相较于 DeepGEMM 提升 1.41×，相较于 FlashInfer 提升 1.13×。
*   **泛化能力**：通过 10–24 分钟的一次性分析，无需任何模型特有调优即可在未知架构上达到最优性能。

#### 7. 潜在局限
*   **硬件依赖**：当前模型基于 Hopper (SM 90a) 架构设计，若迁移至 Blackwell 架构需重新进行少量的配置分析。
*   **通信开销**：该方案侧重于单 GPU 内的计算内核调度，对于多卡环境下的专家并行通信（Expert-parallel communication）开销未纳入优化范围。
*   **极端场景**：在极小批（$S=1$）或极度均匀的路由场景下，动态调度带来的收益会收敛于静态调度。
